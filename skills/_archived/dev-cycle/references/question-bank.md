# Extended Question Bank

Deep-dive questions organized by domain. Use these when the 8 mandatory questions need more depth, or when the user's answers are too surface-level.

## Problem Exploration

- What triggered this idea? Was there a specific incident or frustration?
- How long have you/users been dealing with this problem?
- What's the cost of NOT solving this? (time, money, frustration)
- Is this a symptom of a deeper problem? (5 Whys candidate)
- Have you tried solving this before? What happened?

## User Understanding

- Walk me through a day in the user's life — where does this problem show up?
- How tech-savvy are the users? What's their tolerance for complexity?
- How many users are affected? (1 person vs. a team vs. thousands)
- What would users do if this solution disappeared tomorrow?
- Are there different user types with different needs?

## Scope Boundary Probing

- What's the simplest version that would still be useful?
- If you only had 2 hours, what's the one thing you'd build?
- What features do similar products have that you explicitly DON'T want?
- Is there a phase 2 or v2 you're imagining? What belongs there, not here?
- What "nice to haves" are you already thinking about? (capture for Won't-Have list)

## Technical Risk Assessment

- What parts of the codebase does this touch? Are any of those areas fragile?
- Are there external dependencies (APIs, services, databases) that could cause issues?
- What's the worst-case performance scenario?
- Is there existing code that does something similar? Can we reuse it?
- What happens if this fails in production? Is it recoverable?

## Success Validation

- How will you demo this to someone? What would you show them?
- What's the minimum bar — if it does [X], it's a success even if nothing else works?
- What metric would change if this works? (even informal — "I'll spend less time on Y")
- How will you know in a week/month whether this was worth building?
- Can you write the release notes now? (forces clarity on what ships)

## Design Trade-offs

- Speed vs. quality — which matters more for this specific feature?
- Build vs. buy — is there an existing tool/library that does 80% of this?
- Simple now vs. flexible later — does this need to evolve, or is it a one-shot?
- User experience vs. development speed — where's the line?
- Consistency with existing code vs. using a better approach — which wins?

## Jobs-to-be-Done Deep Dive

- When did you first realize you needed this? What were you doing?
- What's the "hiring criteria"? What makes a solution good enough to "hire"?
- What would make you "fire" this solution and go back to the old way?
- Are there social/emotional aspects? (looking professional, reducing anxiety, etc.)
- What's the "competing" solution — including doing nothing?
